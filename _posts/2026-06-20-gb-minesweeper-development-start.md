前回は、AIと一緒に開発を進めるための `PROJECT.md` とWBSの運用方法を決めました。

今回は実際にマインスイーパー開発を開始し、ゲームボーイソフトを作るための開発基盤を構築します。

## RGBDSの開発環境を決める

RGBDSは現在使用している **1.0.1系** を開発基準としました。

個人開発では古いバージョンへの互換性よりも、自分の環境で確実にビルドできることを優先します。

## ディレクトリ構成

まずは必要最小限の構成にしました。

```text
gb-minesweeper/
├── README.md
├── PROJECT.md
├── Makefile
├── src/
├── include/
├── assets/
├── tools/
├── obj/
└── build/
```

将来RPG化する可能性はありますが、使わないディレクトリは最初から作らない方針です。

## buildとobjの役割

今回一番悩んだのが生成物の配置でした。

最終的には次のように整理しました。

- **obj** : ビルド途中で生成される中間ファイル
- **build** : 最終成果物

例えば、

- `.o`
- `.2bpp`
- デバッグ用生成物

などは `obj` に配置します。

一方、

- `.gb`

のみを `build` に配置します。

このように整理することで、Gitで管理するものと生成物の役割が明確になりました。

## Makefileを作成

Makefileでは次のターゲットを用意しました。

- `make`
- `make clean`
- `make run`

また、環境依存となるエミュレータのパスは環境変数 `EMULATOR` で指定する設計にしました。

これにより、GitHubへ公開しても利用者は自分の環境に合わせて実行できます。

## WSL2からSameBoyを起動

開発はWSL2上で行っていますが、エミュレータはWindows版SameBoyを使用しています。

WSLからWindowsアプリケーションはそのまま起動できます。

```bash
export EMULATOR="/mnt/c/<SameBoyのインストールパス>/sameboy.exe"
```

※ `<SameBoyのインストールパス>` は各自の環境に合わせて変更してください。

設定後、

```bash
make run
```

でROMを起動できることを確認しました。

## 最小ROMを作成

まずはRGBDSでビルドできる最小構成を作成しました。

- エントリポイント
- メインループ
- Makefileによるビルド
- rgbfixによるROMヘッダ設定

画面にはNintendoロゴが表示されたままですが、これは画面を書き換える処理をまだ実装していないためで、現時点では期待どおりの動作です。

## rgbfixでROMヘッダを設定

ROMヘッダの詳細はMakefileから `rgbfix` を利用して設定します。

PROJECT.mdには

- ROMヘッダはrgbfixで設定する
- カートリッジタイプはROM ONLY

という設計だけを記載し、具体的なオプションはMakefileを正本とすることにしました。

## 今回学んだこと

今回の作業では、単に最小ROMを作るだけではなく、

- 設計はPROJECT.md
- 実装はソースコード・Makefile
- 履歴はGit

という役割分担を意識しました。

また、PROJECT.mdには設計だけを書き、実装の詳細は実際のファイルを正本にすることで、情報の重複を避けられることを実感しました。
