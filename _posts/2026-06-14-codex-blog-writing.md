## 前提

この記事では、以下の環境を前提にします。

- WSL2上にJekyllブログ環境がある
- ブログのディレクトリは `~/gbdev-blog`
- Codex CLIをWSL2にインストールして使う
- ChatGPTでCodex向けプロンプトを作成し、それをCodexに貼り付けて作業する
- Codexの作成内容は必ず人間が確認する

Codexはファイル作成や修正を実行できます。
そのぶん、出力された記事や差分をそのまま信用せず、自分で確認することが大事です。

## Codexを使う理由

ChatGPTとCodexは、同じように文章作成や相談に使えます。
ただし、ブログ作成では役割を分けたほうが進めやすいと感じました。

ChatGPTは、記事の方針整理やプロンプト作成に向いています。
たとえば「この記事では何を書くか」「どの順番で説明するか」「Codexに何を依頼するか」を考える作業です。

一方、Codexは実際のファイル作成・修正・diff確認に向いています。
JekyllブログのディレクトリでCodexを起動すると、既存ファイルを読んだうえで、新しいMarkdownファイルを作成したり、既存ページを修正したりできます。

ブログ記事作成では、まずChatGPTで「何を書かせるか」を整理し、それをCodexに渡すと進めやすくなります。
いきなりCodexに丸投げするより、ファイル名、Front Matter、見出し、画像パス、書き方の注意を細かく指定したほうが失敗が少なくなります。

## Codex CLIのインストール

Codex CLIを使うために、まずNode.jsを用意しました。
ここでは `nvm` を使ってNode.jsをインストールします。

### nvmのインストール

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/master/install.sh | bash
source ~/.bashrc
nvm --version
```

`nvm --version` でバージョンが表示されれば、`nvm` の準備は完了です。

### Node.jsのインストール

次に、LTS版のNode.jsをインストールします。

```bash
nvm install --lts
node -v
npm -v
```

`node -v` と `npm -v` でバージョンが表示されることを確認します。

### Codex CLIのインストール

Node.jsとnpmが使えるようになったら、Codex CLIをインストールします。

```bash
npm install -g @openai/codex
codex --version
```

`codex --version` でバージョンが表示されれば、Codex CLIを起動できる状態です。

## ブログディレクトリでCodexを起動する

Codexには、作業したいディレクトリで起動してもらいます。
今回の場合はJekyllブログのディレクトリへ移動してから起動しました。

```bash
cd ~/gbdev-blog
codex
```

初回起動時にはサインインが必要でした。
今回はChatGPTアカウントでサインインしました。

認証時にはブラウザでログイン画面を開き、案内に従ってログインします。
ここで表示される認証URLの全文や、ローカル環境のユーザー名、PC名などはブログ記事には書かないようにします。

また、起動時にディレクトリを信頼するか聞かれました。
今回は自分のブログ作業用ディレクトリなので、信頼して続行しました。

## まずは読み取り専用で分析させる

最初からファイルを変更させるのではなく、まずは現在のJekyllブログを分析してもらいました。

依頼内容は以下のような形です。

```text
現在のJekyllブログを分析してください。
改善点を提案してください。
ファイルは変更しないでください。
```

このように「ファイルは変更しないでください」と明記すると、まずは読み取り専用の調査として進められます。

このとき、以下のような改善提案が出ました。

- `about` ページが初期状態だった
- `_config.yml` に `lang` と `timezone` がなかった
- 404ページが英語のままだった
- `index` ページに導入文がなかった
- 投稿本文のH1がFront Matterのタイトルと重複していた
- 未追跡ファイルがあることも指摘された

最初に分析だけさせると、いきなり修正される心配が少なく、どこから直すかを人間側で選びやすくなります。

## 低リスクな修正だけCodexに依頼する

分析結果を見たあと、影響範囲が小さい修正だけCodexに依頼しました。

実際には、以下のように修正範囲を限定しました。

```text
以下の低リスクな改善だけ実施してください。

- _config.yml に lang: ja と timezone: Asia/Tokyo を追加
- about.markdown を日本語のブログ紹介文に修正
- 404ページを日本語化
- index.markdown に短い導入文を追加
- _posts/2026-06-07-jekyll-setup.md の本文側H1を削除し、## はじめに から始まる形に修正

GemfileやGitHub Pagesのビルド方式は今回は変更しないでください。

修正後に git diff を表示してください。
可能なら bundle exec jekyll build も実行してください。
```

ここで大事なのは、修正対象を明確に指定することです。
今回は固定ページや設定の軽い修正に絞り、GemfileやGitHub Pagesのビルド方式のように影響範囲が大きいものは変更しないように指定しました。

修正後は `git diff` を表示してもらい、どのファイルがどう変わったかを確認しました。
さらに、可能であれば `bundle exec jekyll build` も実行して、Jekyllとしてビルドできるか確認します。

## 自分でも確認する

Codexが修正したあとも、自分で確認します。
特にブログ記事や設定ファイルは、見た目や文章の意味まで確認する必要があります。

まず、差分を確認します。

```bash
git diff
```

次に、Jekyllのビルドが通るか確認します。

```bash
bundle exec jekyll build
```

ローカルで表示確認する場合は、開発サーバーを起動します。

```bash
bundle exec jekyll serve
```

ブラウザで表示を確認し、問題がなければコミットします。

```bash
git status
git add .
git commit -m "ブログの基本設定と固定ページを整備"
```

Codexに修正を任せても、最終確認とコミット判断は人間が行います。

## 記事作成をCodexに依頼する

実際に記事2、記事3を作るときは、まずChatGPTでCodex向けプロンプトを作成しました。
そのプロンプトをCodexへ貼り付けて、Jekyll用のMarkdownファイルを作成してもらいました。

記事作成を依頼するときは、以下を具体的に指定しました。

- 作成するMarkdownファイル名を指定する
- Front Matterを明確に指定する
- 記事に含める内容を箇条書きで指定する
- 画像パスを指定する
- 画像参照では `{{ site.baseurl }}` を使うように指定する
- 書き方の注意を指定する
- 作業完了後に報告してほしい内容も指定する

たとえば、Jekyllの記事ではFront Matterの形式が重要です。
`layout`、`title`、`date`、`category`、`tags` を最初から指定しておくと、Codexが既存記事に近い形で作成しやすくなります。

画像を使う場合も、パスを具体的に指定します。
GitHub Pagesで公開することを考えると、記事内の画像参照は以下のように `{{ site.baseurl }}` を使う形にしておくと安心です。

```markdown
![画像の説明]({{ site.baseurl }}/assets/images/example/image.png)
```

「コードブロックに不要なid属性を付けない」「個人情報を書かない」「全文ではなく抜粋だけ載せる」といった注意も、最初に書いておくと修正の手間が減ります。

## Codexで起きた修正例

Codexで記事を作成したあと、内容が実際の作業とずれていたため修正したこともありました。

たとえば、最初の記事案では「HELLO表示」と書かれていました。
しかし実際には、ゲームボーイ画面に `HELLO` という文字列を表示したわけではありません。
このときの `hello` は、プロジェクト名やROMファイル名として使っていただけでした。

そのため、記事の内容を以下のように修正しました。

- タイトルを「RGBDSで最初のゲームボーイROMを作る」に変更した
- ファイル名も内容に合わせて変更した
- 「HELLO表示成功」ではなく「正常起動を確認した画面」という画像キャプションに修正した
- `hello` はプロジェクト名・ROMファイル名であり、表示文字列ではないと説明した

このように、AIは文脈からもっともらしい文章を作れますが、実際に起きたこととずれる場合があります。
特に技術記事では、コマンド、ファイル名、画像の意味、実行結果が事実と合っているかを必ず確認する必要があります。

## Codexに依頼するときのコツ

Codexにブログ作業を依頼するときは、以下を意識すると進めやすくなります。

- 最初は「ファイルは変更しないでください」と指定して分析だけさせる
- 修正範囲を限定する
- 変更してほしくないファイルも明記する
- ファイル名、Front Matter、画像パスを具体的に書く
- `git diff` を表示させる
- `bundle exec jekyll build` や `bundle exec jekyll serve` で確認する
- AIが書いた記事の事実関係は必ず確認する
- 画像パスは `{{ site.baseurl }}` を使う
- 個人情報やローカル環境のユーザー名を書かせない

Codexは便利ですが、すべてを自動で正しく判断してくれるわけではありません。
「何をしてほしいか」と「何をしてほしくないか」を両方書くと、意図に近い結果になりやすくなります。

## まとめ

今回は、ChatGPTとCodex CLIを使ってJekyllブログの記事作成・修正を進める流れをまとめました。

ChatGPTでは、記事の方針やCodex向けプロンプトを作ります。
Codexでは、Jekyllブログのファイル作成・修正を行います。

修正後は `git diff` で差分を確認し、`bundle exec jekyll build` でビルドできるか確認します。
必要に応じて `bundle exec jekyll serve` で表示も確認します。

最終的な事実確認は人間が行います。
ChatGPTとCodexを分担させると、ブログ作成の流れを整理しながら進めやすくなりました。
